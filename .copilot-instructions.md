# Copilot Instructions - Audio DSP with Teensy Portability

## Core Philosophy

This project is designed to be **simultaneously a professional VST plugin AND a Teensy hardware implementation**. Every decision prioritizes Teensy feasibility without compromising VST quality.

---

## CRITICAL CONSTRAINTS (Non-Negotiable)

### 1. Teensy Feasibility
**Why:** Same codebase must work on Teensy 4.1 without modification.

**Rules:**
- ✓ Pure C++17 (no C++20 features)
- ✓ `std::atomic` for thread-safe parameters
- ✓ Stack buffers: `float buffer[4096]`
- ✓ Pre-allocated vectors with `reserve()`
- ✗ NO external libraries in `src/dsp/`
- ✗ NO `std::vector` modifications in audio callbacks
- ✗ NO `std::unique_ptr` if it requires allocation during real-time
- ✗ NO `new`/`delete` in `processBlock()`

**Example:** ✓ GOOD
```cpp
// In prepare() - once per session
void prepare(int sampleRate) {
    buffer_.reserve(48000 * 4);  // Pre-allocate capacity
}

// In processBlock() - called repeatedly
void processBlock(float* out, int numSamples) {
    for (int i = 0; i < numSamples; ++i) {
        out[i] = buffer_[readPos_++];  // No allocation
    }
}
```

**Example:** ✗ BAD (violates Teensy)
```cpp
void processBlock(float* out, int numSamples) {
    std::vector<float> temp(numSamples);  // ALLOCATION IN REAL-TIME!
    // ...
}
```

### 2. Teensyduino Translatability

**Why:** Want 1:1 source code parity between VST and Teensy.

**Rules:**
- Functions must compile identically on Teensy with no #ifdefs
- No Teensy-specific includes in `src/dsp/`
- Teensy-specific code goes in `src/hardware/teensy/` only
- Memory model must match: use `float*` pointers, not JUCE's `AudioBuffer`

**Example Structure:**
```
src/dsp/CircularBuffer.cpp          ← Works on VST + Teensy
src/juce/PluginProcessor.cpp        ← VST-only wrapper
src/hardware/teensy/I2SDriver.cpp   ← Teensy-specific I/O
```

---

### 3. Real-Time Audio Safety

**Why:** Audio callbacks run at millisecond intervals with no time for glitches.

**Forbidden in `processBlock()` and audio callbacks:**
- ✗ Memory allocation: `new`, `malloc`, `std::vector::push_back`
- ✗ Locking: `std::mutex`, `std::lock_guard`
- ✗ Blocking I/O: file reads, network calls
- ✗ Virtual function calls (except inlined)
- ✗ Exceptions
- ✗ Printf/logging (on Teensy audio thread)

**Allowed in audio callbacks:**
- ✓ `std::atomic` loads/stores
- ✓ Simple arithmetic and loops
- ✓ Inline functions
- ✓ Pre-allocated buffer access
- ✓ Non-blocking parameter queries

**Example:** ✓ GOOD (real-time safe)
```cpp
void processBlock(float* out, int numSamples, double tempo) {
    float speed = playbackSpeed_.load();  // Atomic, instant
    for (int i = 0; i < numSamples; ++i) {
        out[i] = calculateSample(tempo, speed);
    }
}
```

**Example:** ✗ BAD (not real-time safe)
```cpp
void processBlock(float* out, int numSamples, double tempo) {
    std::lock_guard<std::mutex> lock(mutex_);  // BLOCKS!
    // ... danger of deadlock/glitch
}
```

### 4. Architecture & File Organization

**Tier Structure:**

```
Tier 1: Foundation (No dependencies)
├── src/dsp/AudioBuffer.h/cpp                Pure C++, portable

Tier 2: DSP Modules (Pure C++, no JUCE, no Teensy specifics)
├── src/dsp/CircularBuffer.h/cpp              Ring buffer, freeze logic
├── src/dsp/Sampler.h/cpp                     Sample playback + tempo sync
├── src/dsp/SamplerBank.h/cpp                 4 samplers container
├── src/dsp/Mixer.h/cpp                       Mode routing (parallel/sequential)
├── src/dsp/FreezeEffect.h/cpp                Orchestrates freeze + stutter

Tier 3: Non-Real-Time (File I/O allowed here)
├── src/audio/AudioFileLoader.h/cpp           WAV, MP3, FLAC loading
├── src/audio/PresetManager.h/cpp             Save/load with embedded samples

Tier 4: Platform Integration
├── src/juce/PluginProcessor.h/cpp            VST3/AU plugin wrapper
├── src/juce/PluginEditor.h/cpp               UI (VST only)
├── src/hardware/teensy/...                   Teensy I2S, MIDI, UI drivers

```

**Golden Rule:**
- Lower tiers = pure C++, no dependencies
- Higher tiers = can use platform-specific libraries
- **Never import from higher tier to lower tier**

## When to Refuse/Question

### Refuse If:
1. Suggestion allocates memory in `processBlock()`
2. Uses `std::mutex` or `std::lock_guard` in audio callback
3. Modifies `std::vector` in real-time path
4. Calls file I/O in audio thread
5. Uses JUCE types in `src/dsp/`
6. Uses virtual functions in audio inner loop
7. Requires Teensy-specific includes in DSP core

### Ask For Clarification If:
1. "Is this called from processBlock()?"
2. "What's the frequency of this call - real-time or UI thread?"
3. "Can this be pre-allocated in prepare()?"
4. "Should this be in src/dsp/ or src/audio/?"
5. "Does this need to compile on Teensy?"

## Memory Constraints

### Teensy 4.1
- **On-chip RAM:** 512 KB (use for audio buffers)
- **PSRAM:** 8 MB (use for samples/presets)
- **Stack size:** ~200 KB (audio callback stack is limited)

### Guidelines
- Freeze buffer default: 4 seconds @ 48kHz = ~192 KB (fits on-chip)
- Max 4 samples @ 5 sec each = 960 KB total (fits in PSRAM)
- Avoid large local arrays on stack; use class members

### Example:
```cpp
// ✓ GOOD - Stack-allocated small buffer
void processBlock(int numSamples) {
    float tempBuffer[256];  // Small, stack-friendly
}

// ✓ GOOD - Pre-allocated class member
class Sampler {
    std::vector<float> sampleData_;  // Allocated once in setSampleData()
};

// ✗ BAD - Dynamic allocation in real-time
void processBlock(int numSamples) {
    std::vector<float> temp(numSamples);  // Memory allocated!
}
```

## Project-Specific Rules

### For This Sampler Plugin
1. **Samples:** One-shot only (no polyphony per sample)
2. **Freeze buffer:** User-configurable, min 1 bar, max 8 bars
3. **MIDI mapping:** C3-C#3 map to samples 0-3
4. **Tempo sync:** Via host BPM (VST) or knob input (Teensy)
5. **Routing modes:** Sequential (input+sampler mixed) or Parallel (input isolated)

### DSP Assumptions
- Sample rate: 44.1 kHz to 192 kHz
- Block size: 64 to 4096 samples
- Channels: 1 (mono) or 2 (stereo) internally
- Freeze buffer: Allocated once, never reallocated

## Reference: src/dsp/ API Requirements

All functions in `src/dsp/` must match this pattern:

```cpp
// ✓ GOOD - Pure C++, no dependencies, real-time safe
class MyDSPModule {
public:
    // Setup (non-real-time, can allocate)
    void prepare(int sampleRate, int maxBlockSize);
    
    // Processing (real-time, zero allocation)
    void processBlock(float* outAudio, int numSamples);
    
    // Parameters (real-time safe via atomics)
    void setParameter(float value);
    float getParameter() const;
};
```

## Summary

**The Golden Rules:**
1. `src/dsp/` = pure C++17, no external deps, no allocations in audio callbacks
2. Use `std::atomic` for thread-safe parameters, never locks in real-time
3. Pre-allocate everything in `prepare()`, modify nothing in `processBlock()`
4. When in doubt, ask: "Is this Teensy-portable and real-time safe?"
5. Document memory usage and constraints

**If you're proposing code and unsure, ask these questions first.**
